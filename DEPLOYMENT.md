# Barnabas Dental — Deployment Guide

## Repository layout

**Canonical paths for Render and Vercel** (repo root):

| Component | Path |
|-----------|------|
| Django API | `backend/` |
| React frontend | `frontend/` |

A legacy local copy may exist at `barnabas-dental-system/backend/` and `barnabas-dental-system/frontend/`. Those are **not** used by Render or Vercel — always deploy from the root `backend/` and `frontend/` directories.

---

## Backend (Render)

1. Connect the repo and apply `render.yaml` (`rootDir: backend` for web + cron services).
2. Set environment variables in the Render dashboard (see checklist below).
3. `DEBUG` is set to `False` in `render.yaml`.
4. Migrations run via `preDeployCommand` and as a fallback in `buildCommand` (for free tier when Shell is unavailable).
5. Create an admin user once after first deploy (see [Local migrate against Aiven](#local-migrate-against-aiven-no-render-shell)).

### Render environment variable checklist

| Variable | Required | Example / notes |
|----------|----------|-----------------|
| `SECRET_KEY` | Yes | Auto-generated by `render.yaml` for the web service; **reuse the same value** on all cron jobs |
| `ALLOWED_HOSTS` | Yes | `barnabas-api.onrender.com` (your Render API hostname, no `https://`) |
| `DATABASE_URL` | Yes | Aiven PostgreSQL URI — see [DATABASE_URL notes](#database_url-notes) |
| `DB_SSL_REQUIRE` | Yes | `True` (set in `render.yaml`; keep enabled for Aiven) |
| `CORS_ALLOWED_ORIGINS` | Yes | `https://your-app.vercel.app` (Vercel frontend URL, no trailing slash) |
| `FRONTEND_URL` | Yes | Same Vercel URL as above |
| `CSRF_TRUSTED_ORIGINS` | Recommended | Same as `CORS_ALLOWED_ORIGINS` if not set, Django defaults to CORS list |
| `EMAIL_HOST` | For reminders | e.g. `smtp.gmail.com` |
| `EMAIL_PORT` | Set in yaml | `587` |
| `EMAIL_USE_TLS` | Set in yaml | `True` |
| `EMAIL_HOST_USER` | For reminders | Gmail address or SMTP user |
| `EMAIL_HOST_PASSWORD` | For reminders | Gmail app password (not your login password) |
| `DEFAULT_FROM_EMAIL` | For reminders | Sender address shown to recipients |

### DATABASE_URL notes

- Use the **Service URI** from the Aiven console.
- SSL is required. The query string must use `sslmode=require` (equals sign), **not** `sslmode-require` (hyphen). A common dashboard typo breaks the connection.
- Example shape (replace with your values from Aiven — **never commit real credentials**):

  ```
  postgres://USER:PASSWORD@HOST:PORT/DATABASE?sslmode=require
  ```

- If a password was ever pasted into chat, committed, or logged, **rotate it in the Aiven console** and update `DATABASE_URL` on Render and in any local `.env`.

### Scheduled jobs (cron)

| Command | Schedule | Purpose |
|---------|----------|---------|
| `expire_pencil_bookings` | Every 15 min | Release expired 4-hour pencil holds |
| `send_appointment_reminders` | Daily 9:00 AM | 24-hour appointment reminders |
| `send_follow_up_reminders` | Daily 10:00 AM | Follow-up after completed visits (7 days) |

Cron services are defined in `render.yaml`. Each cron job must share `DATABASE_URL` and `SECRET_KEY` with the web service. Reminder cron jobs also need `EMAIL_HOST`, `EMAIL_HOST_USER`, `EMAIL_HOST_PASSWORD`, and `DEFAULT_FROM_EMAIL`.

## Frontend (Vercel)

1. Set the project **root directory** to `frontend`.
2. Set `VITE_API_URL` to your Render API base **including the `/api` suffix**:

   ```
   VITE_API_URL=https://barnabas-api.onrender.com/api
   ```

   Do **not** use a placeholder like `https://api.example.com`. The frontend appends paths to this base (e.g. `/users/token/` → `https://barnabas-api.onrender.com/api/users/token/`).

3. Redeploy after changing env vars (Vite bakes `VITE_*` at build time).
4. `vercel.json` handles SPA routing.

---

## Database (Aiven)

1. Enable automated backups in the Aiven console.
2. Migrations run on Render deploy (`preDeployCommand` + `buildCommand` fallback).
3. For one-off tasks when Render Shell is unavailable, run commands locally against Aiven (below).

### Local migrate against Aiven (no Render Shell)

Render’s free tier may not offer Shell. Run migrations and `createsuperuser` from your machine:

```bash
cd backend
cp .env.example .env   # if you don't have one yet
```

In `.env` for this session only (do **not** commit):

```env
DEBUG=False
DATABASE_URL=postgres://USER:PASSWORD@HOST:PORT/DATABASE?sslmode=require
DB_SSL_REQUIRE=True
```

Do **not** set `DB_NAME` / `DB_HOST` etc. when using `DATABASE_URL` — use one approach only.

```bash
pip install -r requirements.txt
python manage.py migrate
python manage.py createsuperuser
```

Unset `DATABASE_URL` afterward if you normally use local `DB_*` vars for day-to-day dev.

## Post-deploy verification

Replace hostnames with yours. Expect JSON from API routes and `200`/`302` from admin.

```bash
# API health — should return JSON (401 without auth is OK)
curl -s -o /dev/null -w "%{http_code}" https://barnabas-api.onrender.com/api/users/me/

# Admin login page — should return 200
curl -s -o /dev/null -w "%{http_code}" https://barnabas-api.onrender.com/admin/login/

# CORS preflight (replace Vercel origin)
curl -s -I -X OPTIONS \
  -H "Origin: https://your-app.vercel.app" \
  -H "Access-Control-Request-Method: POST" \
  https://barnabas-api.onrender.com/api/users/token/
```

From the browser: open the Vercel app, try login/register, and confirm network requests go to `https://barnabas-api.onrender.com/api/...` not `api.example.com`.

## Media / profile pictures (Phase 1)

Profile pictures are stored on the server filesystem under `backend/media/avatars/`.

### Local development

- `MEDIA_URL` and `MEDIA_ROOT` are configured in `backend/config/settings.py`.
- Uploaded files are served at `/media/...` when `DEBUG=True`.

### Render (important)

Render’s filesystem is **ephemeral**:

- Uploaded avatars survive until the next **deploy** or **service restart**, then they are **lost**.
- Phase 1 serves `/media/` from the app container (see `config/urls.py`). No extra env vars are required.
- For production-grade persistence, plan Phase 2: object storage (e.g. S3, Cloudinary) with `django-storages`.

### API

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `POST` | `/api/users/me/avatar/` | Upload (`multipart/form-data`, field `avatar`) |
| `DELETE` | `/api/users/me/avatar/` | Remove custom avatar |

Allowed types: JPG, PNG, WebP. Max size: 5 MB.

## Security

- **Never commit** `.env` files or real passwords. Only `.env.example` files belong in git.
- SMTP and database credentials are **environment variables only** — not stored in the database or audit logs.
- If `DATABASE_URL` or any Aiven password was exposed, rotate the password in Aiven and update Render (and local `.env` if used).

## Local development

Uses local PostgreSQL via `DB_*` vars — **do not set `DATABASE_URL`** in `.env` for normal local dev.

```bash
# Backend
cd backend
cp .env.example .env
python manage.py migrate
python manage.py runserver

# Frontend
cd frontend
cp .env.example .env
npm install
npm run dev
```

Manual pencil expiry (local): `python manage.py expire_pencil_bookings`

## Deploy readiness checklist

Run these from the **repo root** (`backend/` and `frontend/`) before pushing:

```bash
# Backend
cd backend
python manage.py check --deploy          # expect warnings only when DEBUG=True locally
python manage.py makemigrations --check --dry-run   # expect "No changes detected"
python manage.py test users appointments billing
DEBUG=False SECRET_KEY=<random-50+-chars> python manage.py collectstatic --noinput

# Frontend (canonical path — not barnabas-dental-system/frontend)
cd ../frontend
npm install
npm run build
# On Vercel, set VITE_API_URL before build:
# VITE_API_URL=https://barnabas-api.onrender.com/api npm run build
```

### Verified locally (pre-deploy)

| Check | Status |
|-------|--------|
| Django tests (`users`, `appointments`, `billing`) | 42 passed |
| `makemigrations --check` | No pending migrations |
| `collectstatic` (DEBUG=False) | OK |
| WSGI loads (DEBUG=False) | OK |
| Frontend `npm run build` | OK |
| `package-lock.json` committed | Yes (required for Vercel) |
| No merge conflicts / no `.env` in repo | OK |
| `render.yaml` migrations (build + preDeploy) | Configured |
| `DATABASE_URL` sslmode typo guard in settings | Configured |

### First deploy order

1. **Aiven** — database running; note Service URI with `?sslmode=require`.
2. **Render** — connect repo; set `DATABASE_URL`, `ALLOWED_HOSTS`, `CORS_ALLOWED_ORIGINS`, `FRONTEND_URL` **before** first deploy (build runs migrations and needs DB).
3. **Render cron jobs** — copy `SECRET_KEY` and `DATABASE_URL` from web service; add SMTP vars to reminder crons.
4. **Vercel** — root directory `frontend`; set `VITE_API_URL=https://barnabas-api.onrender.com/api`; redeploy after any env change.
5. **Admin user** — `python manage.py createsuperuser` locally against Aiven (see above); assign `admin` role in Django admin → User roles if needed.
6. **Post-deploy** — run curl checks in [Post-deploy verification](#post-deploy-verification).

### Common pitfalls

| Symptom | Fix |
|---------|-----|
| API 400/502 on deploy | `DATABASE_URL` typo (`sslmode=require` not `sslmode-require`) |
| Frontend calls `api.example.com` | Set `VITE_API_URL` on Vercel with `/api` suffix; redeploy |
| CORS errors | `CORS_ALLOWED_ORIGINS` must exactly match Vercel URL (`https://…`, no trailing slash) |
| 403 on admin Create Staff | Log in at `/admin` as superuser with `admin` clinic role; use repo `frontend/`, not `barnabas-dental-system/frontend` |
| Render build fails on migrate | Set `DATABASE_URL` in dashboard before triggering deploy |
